1. Estado y límites de la validación

No afirmar “validado visualmente”. La instancia local no está disponible: Supervisor apunta a /app/frontend y /app/backend, pero este repositorio es una aplicación Vite en /app. Además, no existe /app/memory/test_credentials.md, faltan variables Supabase y no se encontró una URL desplegada accesible. Por las reglas del proyecto no se creó una sesión ni se simuló autenticación.

Por tanto, la evidencia de este documento es estática: estructura React, CSS, contratos de rutas, tokens, accesibilidad y comportamiento deducible. Los puntos visuales e interactivos se incluyen como pruebas obligatorias posteriores.

2. Mapa de implementación actual

PC: Sidebar

  • Visible desde 1024px.
  • Ancho expandido 248px; colapsado 76px.
  • Rutas: Reclutamiento, Reporte Diario, Candidatos, Plantilla, Vacantes y Features.
  • Incluye estado activo, tooltips colapsados, usuario, tema y cierre de sesión.

Móvil/tablet: Header + BottomTabBar

  • Visibles hasta 1023px.
  • Header: marca y avatar.
  • Barra: Reclutamiento, Candidatos y botón Menú.
  • Sheet: Plantilla, Reporte Diario, Vacantes, Features, tema y salida.

Breakpoint: la pareja max-width: 1023px / min-width: 1024px evita solapamiento CSS. Sin embargo, el componente móvil permanece montado en React y eso genera un bug de estado al redimensionar con el sheet abierto (P0 abajo).

3. Aspectos correctos que deben preservarse

Responsive y geometría

  • La navegación usa un único cambio de modo en 1024px.
  • El contenido desktop comparte el ancho de sidebar mediante --_app-sidebar-w.
  • Existe reserva inferior para que la barra flotante no tape el final del contenido.
  • Safe area superior/inferior está contemplada.

Semántica e interacción

  • aside, nav, listas, enlaces y botones nativos.
  • Botones de icono con nombre accesible.
  • Sheet con Escape, bloqueo de scroll, focus trap y restauración de foco.
  • Estado activo no depende solo del color: usa indicador o fondo.

Sistema visual

  • La mayor parte de colores, spacing, radios, sombras y tipografía usa tokens.
  • Targets principales de navegación usan --touch-target-min: 48px o --mobile-nav-height: 56px.
  • Sidebar plano con hairline; navbar flotante con elevación tokenizada.

Robustez móvil

  • Uso de VisualViewport para reducir salto de la barra en iOS.
  • Textos de sesión y navegación truncan o se distribuyen intencionalmente.
  • El sheet usa 100dvh y contenido interno desplazable.

4. Correcciones requeridas, por prioridad

P0 — Cerrar/desmontar el menú móvil al entrar a desktop

Bug funcionalResponsiveBottomTabBar.tsxBottomTabBar.css

Problema: BottomTabBar siempre permanece montado. Si el sheet está abierto y el viewport cruza de 1023px a 1024px, sheetOpen continúa en true, el overlay fullscreen sigue con display:block porque no está acotado por media query y body.style.overflow continúa en hidden. La Sidebar puede quedar totalmente tapada y la página bloqueada.

Implementación pedida:

  1. Crear o reutilizar una media query compartida para el modo desktop (min-width: 1024px).
  2. Al entrar a desktop, ejecutar setSheetOpen(false); el cleanup existente debe restaurar el overflow y el foco.
  3. Agregar defensa CSS: en desktop, forzar .bottom-sheet-overlay { display: none !important; } aunque el atributo data-open exista.
  4. No cambiar rutas ni lógica de cierre de sesión.

Aceptación: abrir Menú a 1023px, cambiar a 1024px y confirmar Sidebar visible, overlay ausente, body desplazable y foco lógico. Volver a 1023px: el sheet debe permanecer cerrado.

P1 — Cumplir el target táctil mínimo en marca y selector de tema

WCAG 2.2AGENTS §5/§6Header.cssThemeToggle.css

Problema: el selector de tema mide 52×28px; su caja interactiva queda por debajo del mínimo. El enlace de marca del Header contiene un logo de 26px y no define un área mínima de 44×44/48×48.

Implementación pedida: conservar la apariencia del switch, pero ampliar su hit area sin deformar el carril (wrapper/botón de 48px mínimo con pseudo-elemento o estructura interna). Dar al enlace de marca min-width/min-height basado en --touch-target-min, manteniendo alineación visual. No usar números nuevos si existe token.

Aceptación: el rectángulo de interacción computado de ambos controles es al menos 48×48px en 390px y 768px; no cambia la altura visual del Header ni se mueve el logo.

P1 — Reparar focus y elevación del popover de usuario

Tokens inexistentesKeyboardSidebar.css

Problema: .sidebar__user-trigger:focus-visible usa var(--color-accent), token no definido; .sidebar__popover usa var(--shadow-lg), también no definido. El navegador descarta esas declaraciones, por lo que el foco puede quedar sin señal visible y el popover sin la elevación esperada.

Implementación pedida: usar los tokens existentes del sistema: box-shadow: var(--shadow-focus) para foco y var(--shadow-elevated) o var(--shadow-prominent) para popover, según el nivel documentado. No crear alias innecesarios y no usar color sticker.

Aceptación: al tabular, el trigger tiene indicador visible en claro y oscuro; el popover conserva hairline y una sombra sutil tokenizada.

P1 — Corregir patrón accesible del popover de usuario

ARIATecladoSidebar.tsx

Problema: el contenedor declara role="menu", pero dentro mezcla un role="menuitem" y un role="switch". Tampoco mueve foco al abrir, no cierra con Escape y el cierre por click exterior no restaura explícitamente el foco. Es un popover de acciones, no un menú ARIA completo.

Implementación preferida: simplificar a un popover semántico normal sin role="menu"/menuitem, manteniendo botones nativos y el switch. Al abrir, mover foco al primer control; cerrar con Escape; al cerrar por Escape o click exterior, restaurar foco al trigger. Si se mantiene patrón menu, implementar íntegramente roving focus y roles válidos, pero es una solución más invasiva y no recomendada.

Aceptación: secuencia teclado: trigger → Enter → foco dentro → Tab por acciones → Escape → popover cerrado y foco en trigger.

P1 — Hacer real la reducción de movimiento

prefers-reduced-motionFramer MotionSidebar.tsxBottomTabBar.tsx

Problema: CSS elimina varias transiciones, pero las animaciones inline de Framer Motion (layoutId, entrada/salida de popover, icono de logout y ThemeToggle) siguen ejecutándose.

Implementación pedida: usar useReducedMotion de Framer Motion o una utilidad existente. Con reducción activa, usar transición de duración 0 y evitar spring/scale/rotate; conservar el estado final y toda la funcionalidad.

Aceptación: emular prefers-reduced-motion: reduce; navegación, sheet, popover y tema cambian sin desplazamientos, escalas ni rotaciones animadas.

P2 — Unificar fuente y orden de navegación

ConsistenciaArquitecturaSidebar.tsxBottomTabBar.tsx

Problema: Sidebar y móvil duplican metadatos de rutas y presentan prioridades distintas. Sidebar: Reclutamiento → Reporte Diario → Candidatos → Plantilla → Vacantes → Features. Móvil: Reclutamiento → Candidatos; sheet: Plantilla → Reporte Diario → Vacantes → Features. La división primaria/secundaria puede mantenerse, pero el orden restante no refleja el orden global.

Implementación pedida: extraer una sola configuración tipada (ruta, label, icono, end, prioridad móvil y grupo). Derivar ambos renders sin duplicar rutas. Mantener Reclutamiento y Candidatos como accesos móviles primarios; ordenar el sheet por el orden global de los elementos restantes: Reporte Diario → Plantilla → Vacantes → Features.

Aceptación: ambas superficies exponen exactamente las mismas seis rutas, labels e iconos; solo cambia la agrupación móvil. Ninguna ruta de App.tsx se altera.

P2 — Completar safe areas laterales

iOS landscapeBottomTabBar.cssSidebar.css

Problema: navbar y sheet usan safe areas verticales, pero los gutters laterales usan solo spacing fijo. En dispositivos con notch en landscape, los controles pueden acercarse demasiado al recorte.

Implementación pedida: combinar spacing con --safe-area-left/--safe-area-right mediante max() en la barra, header del sheet, filas y footer. Aplicar también al padding lateral del Sidebar si el producto contempla PWA desktop con inset. No duplicar padding ni introducir overflow.

Aceptación: emular insets laterales; ningún control o texto invade el área insegura y la barra continúa centrada.

P2 — Limpiar código muerto y duplicación local

AGENTS §7/§8Sin cambio UX

Problema: hay imports y variables sin uso en Sidebar y BottomTabBar (user, varios iconos, createPortal, sileo en Sidebar). El bloque Features está duplicado para estado colapsado/expandido. Existen reglas CSS sin markup correspondiente (sidebar__popover-header, sidebar__session-name, badge de documentos inalcanzable).

Implementación pedida: retirar únicamente código confirmado como inalcanzable/no usado y reutilizar el mismo patrón de render ya empleado por NAV_ITEMS para Features. No refactorizar fuera de navegación, no cambiar rutas y no tocar rutas-app.

Aceptación: lint sin imports/variables muertas en estos archivos, markup más corto y comportamiento idéntico.

P2 — Evitar acceso de almacenamiento durante inicialización de render

RobustezAppShell.tsx

Problema: useState(readSidebarCollapsed) accede a window desde el inicializador. Funciona en el Vite cliente actual, pero dificulta SSR, pruebas aisladas y render no-browser.

Implementación pedida: proteger explícitamente con typeof window === 'undefined' o mover la lectura a una utilidad/browser effect sin flash perceptible. Mantener la clave sidebar-collapsed y la persistencia existente.

P2 — Revisar contraste y color primario antes de cambiarlo

Decisión del productoNo automatizar

Observación: desing.md documenta azul #0075de como único acento estructural, pero global.css define --color-primary: #000000. Esta contradicción es global y no debe corregirse como efecto colateral de navegación.

Instrucción: Gemini debe detenerse y pedir confirmación antes de cambiar el token global. En esta tarea solo debe asegurar que la navegación consuma var(--color-primary); no debe decidir si el valor final es azul o negro.

5. Plan de ejecución exacto para Gemini 3.1 Pro High

  1. Preflight obligatorio: leer /app/AGENTS.md, /app/desing.md y este informe. No abrir ni tocar /app/rutas-app.
  2. Preservar contratos: no cambiar autenticación, Supabase, fetch de badge, rutas, localStorage, logout, tema ni contenido de páginas.

  5. Aplicar P2 local: fuente compartida de navegación, safe areas y limpieza de código muerto. Evitar refactor amplio.
